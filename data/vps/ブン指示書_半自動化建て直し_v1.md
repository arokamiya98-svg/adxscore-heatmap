## 0. 一行サマリ
**「常時24h × git-pull 60秒ループ」のwatcherを廃止し、(A) 毎時1回の単発同期ジョブ と (B) トレード時だけ手動起動する前景エンリッチwatcher の2本に分離する。**

---

## 1. なぜ建て直すか（2026-06-24 事故の構造）
旧 `auto_sync_daily.sh`（311行・常駐デーモン）が以下4点の合わせ技でMac全体を詰まらせ、Bashが10分ハング（ADXSCOREもPRIVATEも全Claudeセッション巻き込み）。

1. **60秒ごとの `git pull --rebase --autostash` が常時gitロックを握りにくる**（手動git作業と競合）
2. **二重keepalive**（LaunchAgent `com.aro.adxscore.watcher` ＋ ログイン項目 `ADXSCORE_Watcher.app`）→ auto_sync が2本起動するレース → 互いにgit撃ち合い
3. **gitに内蔵timeoutが無い** → 詰まると無限待ち → Bashツール上限(約10分)で強制帰還
4. CPU飢餓（増幅器）

→ 根本原因は **「常時自動ループ」という形そのもの**。`run_to`（git timeout）は応急で投入済（`auto_sync_daily.sh` `26556de`）だが、本建て直しで形ごと作り直す。

詳細な切り分け（「詰まる(hang=環境の本物)」 vs 「ENOSPC誤報/文字化け(=モデル劣化・再起動案件)」）は [[memory: claude-temp-burst-enospc]]。

## 2. 設計原則（最重要）
**「git pull の常時ループ（VPS受信）」を、系統Aエンリッチの watch から完全に引き剥がす。**

- VPS受信（git pull）＝ **毎時1回の単発ジョブに隔離**。常駐させない。
- 系統Aエンリッチ＝ **ローカルのファイル監視のみ（gitを触らない）**。トレード時だけ手動起動・自動終了。
- 重い自動化は「無人で動く必要があるVPSだけ」。Mac側は「あろさんがいる時に動く」軽いトリガーに落とす。

## 4. 実装するもの

### (A) `hourly_sync.sh`（新規・単発実行・launchdが毎時起動）
**役割**: VPSの毎時pushを受信し、変わったものだけ再生成・公開して**終了する**（常駐しない）。

**処理フロー**:
1. **多重起動ガード**: 自分が既に走っていれば即 exit（pgrep / lockfile いずれか）。
2. `run_to 15 git fetch` → `run_to 45 git pull --rebase --autostash`（`run_to` は `auto_sync_daily.sh` から流用）。失敗/timeoutはログして続行（次の毎時で自己回復）。
3. **日次（②）**: `mt5_data/daily/*.csv` が生成HTML（`data/trades/processed/daily_calendar_v3.html`）より新しければ → `run_daily_calendar.sh --no-open`（既存の鮮度照合ロジックを流用可）。
4. **週次（④）**: wavelog系CSV（`FractalWaveLog_D1_*` / `H4_*` / `H4PhaseAuto_weekly` / `ADX_Weekly_Above_v4` 等、MT5 Files→mt5_data同期込み）が週次生成物より新しければ → `run_pipeline.sh --no-open`。
5. **公開**: 変更があった分だけ `git add <PUBLISH_FILES> && git commit && run_to 30 git push`（push前に `run_to 45 git pull --rebase`）。
6. exit。

**禁止**: while/sleep の常駐ループにしないこと。**1回走って必ず終わる**。スケジューリングはlaunchdに委ねる。

### (B) `update_mani.sh`（新規・系統A・手動）
**役割**: 今日(2026-06-24)おぱが手で回した系統Aの前後事務（prepare→merge→generate→publish）を1コマンドに束ねる。watcherという常駐は使わない。

**2モード**:
- **`./update_mani.sh`（単発）**: iCloud→imports橋渡し → `prepare_trade_input.py`（trade_input生成＋MT5 Files配置）→ ※ここでMT5 Snapshotは**手動ゲート**（スクリプトはユーザーに「MT5でTrade_Snapshot_Builderを回して」と促して待つ or 既に enriched があれば続行）→ MT5出力を mt5_data へ → `prepare_trade_input.py --enriched`（merge）→ `generate_daily_calendar_v3.py` → 公開。
- **`./update_mani.sh --watch`（前景watch・自動終了）**: 前景で起動し、
  - `imports/FX_*.csv` 新着 → prepare → MT5 Files配置 → 「Snapshot回して」と表示
  - `MT5 Files/trades_enriched.csv` 更新検知 → mt5_dataへコピー → merge → generate → 公開 → **1回成功したら自動exit**（または Ctrl+C）。

**安全要件（厳守）**:
- **git pull の常時ループを入れない**。watch対象はローカルファイルのmtimeのみ。
- git操作は**最後の公開pushの1回だけ**（`run_to` 付き、push前に `run_to git pull --rebase`）。
- **前景実行**（デーモン化・nohup常駐禁止）。終わったら消える。多重起動ガードを入れる。

### (C) launchd 単一化（②④用）
- **新plist**: `com.aro.adxscore.hourly`（`~/Library/LaunchAgents/`）
  - `ProgramArguments`: `/bin/bash /Users/aro/Desktop/ADXSCORE/hourly_sync.sh`
  - `StartCalendarInterval`: 毎時 **Minute=10**（VPSの :00 push 完了後に受信するためオフセット）
  - `KeepAlive`: **false**（単発ジョブ。デーモンにしない）
  - `RunAtLoad`: true（ログイン直後にも1回）
  - `WorkingDirectory` / `EnvironmentVariables(PATH,HOME)` は旧plist踏襲
  - `StandardOutPath`/`StandardErrorPath`: `hourly_sync.log`
- **⚠️ TCC検証（必須）**: 旧構成でログイン項目.appが併設されてた理由はおそらく **launchd から ~/Desktop 配下を読む際の TCC ブロック**。実装前に「launchd 起動の `/bin/bash` が `~/Desktop/ADXSCORE` を読めるか」を実機確認すること。ブロックされる場合は (a) `/bin/bash` にフルディスクアクセス付与 か (b) 単一のログイン項目方式（ただし**git-pullループは入れない**単発呼び出しに限る）を選ぶ。**二重keepaliveには戻さない**。

## 5. 受け入れ条件（テスト）
1. `hourly_sync.sh` 単発実行：daily/ 更新あり→日次カレンダー再生成、wavelog更新あり→週次再生成、変更なし→何もせず終了。**いずれもプロセスが残らない**（`ps` で確認）。
2. `hourly_sync.sh` を2つ同時起動→ 2本目が多重起動ガードで即exit。
3. git fetch/pull を擬似ハングさせて（or ネット断）→ timeout で帰る・watcherが死なない。
4. `update_mani.sh --watch` 起動→ FX_*.csv投入→prepare動作→（MT5 Snapshot手動）→ enriched検知→merge→generate→公開→**自動exit**。実行中 `git pull` ループが走らないこと（ログ確認）。
5. launchd 登録後、毎時 Minute=10 に発火しpull→（必要なら）生成→push→終了をログで確認。**常駐プロセスが残らない**。
6. VPS↔Mac 並行で git 事故ゼロ（`git pull --rebase` 規律・push前pull）。
