## 1. 故障モード台帳（今回の全数洗い出し）

### FM-1: 「 2」複製の.git混入 → fetch即死【再発性・致命】
- 混入経路は手作業由来のFinderコピー。`.git`に入った時だけ全動脈が黙って止まる。
- 現状ガードなし。診断: `find .git -name "* 2*"` + hourly_sync.logのfetch失敗連発。

### FM-2: 週次パイプラインが毎時フル再実行【重さの本丸】
- 構造: ②自動集計CSV（ADX_Weekly_Above_v4等）を**VPSが毎時push** → Macがpull → mtime比較（hourly_sync Step3）が毎時発火 → `run_pipeline.sh` フル実行（MT5同期＋python×2＋iCloudコピー＋commit＋push）。
- 実測: **「weekly update」コミットが1日17〜22回**（7/17は22回）。深夜も毎時回る。
- Widget Webの要件は「1日1回更新の週次マクロ」＝**要件の17〜22倍の頻度**で回ってる。

### FM-3: コミット洪水
- VPS 24/日 + weekly 〜22/日 + auto-publish 〜20/日 + mani = **60〜70コミット/日**。
- 総数1122コミット・.git 35MB（今は許容内。ただ増加速度が速い）。

### FM-4: タイムアウトの穴が非対称
- hourly_sync自身のgitは `run_to`（TERM→KILL）で囲ってあるが:
  - `run_daily_calendar.sh` / `run_pipeline.sh` の呼び出しは**タイムアウト無し** → 中でハングするとlock保持のまま、以降の毎時が全部「稼働中→即exit」。
  - `run_pipeline.sh` 内の生 `git push`（Step5）もタイムアウト無し。
- run_to のSIGKILLはgit書き込み中に当たると中間状態（lock残骸等）リスク。

### FM-5: 手動同期リスト（同根の穴・既知）
- PUBLISH_FILES が hourly_sync と run_daily_calendar Step2.6 で二重管理。7/14 d1_env.json置き去り事故が実績。

### FM-6: ログに日付がない
- `ts()` が %H:%M:%S のみ → 障害解析のたびに日付推理が必要（今回も逆算に苦労）。

### FM-7: 異常の検知者が「あろさんの目」しかない
- fetch 88連続失敗しても通知ゼロ。ウィジェットの停滞で人間が気づく構造。7/5のVPS 56h沈黙も同型。

### 健全と確認できたもの
- **VPS push script**: 非対話git（GIT_TERMINAL_PROMPT=0）+ 120sタイムアウト + 失敗は次の毎時が回収、の設計済み（7/5事故の学び）。
- **EA 3系統**: OnTimer毎時・フル再生成（冪等）・FATAL時はその回スキップで次回再試行。構造は良い。
- **多重起動ガード**（mkdir lock）・pull-rebase-before-push 作法は機能してる。

## 2. 構造診断 — なぜ「対策を入れるほど詰まる」のか

これまでの対策は「**部品を足す**」型（watcher→launchd、lock、run_to、監視リスト、手動同期リスト…）。部品が増えるたびに:
1. **手動同期点**が増える（リスト二重管理→7/14事故）
2. **暗黙の前提**が増える（.gitは綺麗なはず→FM-1で崩壊）
3. 一方で**自己修復と異常検知が無い**ので、異物1個で全体が「黙って」止まる

→ 次の設計軸は「足す」ではなく **①冪等・自己修復 ②頻度の適正化 ③異常の可視化** の3本。

## 3. 軽量化設計の叩き台（実装はブン指示書を切ってから）

### 案A: 自己修復ガード【最小・即効・リスクほぼゼロ】
hourly_sync 冒頭に3行級の追加:
- `.git/refs` 配下と `.git` 直下の「* 2*」複製を fetch 前に自動掃除（ログに記録）
- fetch失敗が **3回連続**したら `osascript -e 'display notification'` でMac通知（検知者を人間の目から機械へ）
- `ts()` に日付追加（`%m-%d %H:%M:%S`）

### 案B′: 毎時鮮度は維持したまま、push/コミットを毎時1回に集約【2026-07-21 あろさん判断で差し替え】
- **旧案B（朝夕絞り）は取り下げ**。あろさん要件=「情報鮮度は毎時維持したい」。毎時再生成のCPU実測は数秒/回（reflogで確認）なので頻度自体は重くない。重いのは**毎時2〜3回のpushと2本のコミット**の方。
- あろさん原案「:15/:30/:45に分散して衝突回避」への実測回答: 衝突型失敗（push/pull失敗）は**443回中12回・全て次の毎時で自己回復**。VPS :00 / Mac :10 のオフセットが既に効いてる。3日凍結の実害は衝突ではなく異物混入（FM-1）由来の88連続fetch失敗。→ 分散を増やす（=launchdジョブを増やす）より、**push回数そのものを減らす**方が同じ思想でより効く。
- **実装**: `run_pipeline.sh` に `--no-push` を追加し、hourly_sync からは weekly も daily も**毎時1コミット1push に集約**（daily の `--no-publish` 集約と同じ既存思想を weekly に延長）。手動実行時は従来どおり内蔵push。
- 効果: Mac発コミット 毎時2本→1本（40+/日→24/日）、push窓が毎時1個＝衝突面積も縮む。鮮度は毎時のまま。

### 案C: タイムアウトの穴埋め
- hourly_sync から run_daily_calendar / run_pipeline 呼び出しにも上限（例: 10分）
- run_pipeline 内の生 git push にタイムアウト（VPS側と同じ思想に揃える）

### 案D: 同根の穴の恒久対応
- PUBLISH_FILES を1ファイル（例: `data/vps/publish_list.txt`）に集約し両スクリプトが読む

### 見送り（今はやらない）
- .gitのgc/履歴圧縮（35MBは許容内・年1点検で十分）
- VPS 24/日 pushの削減（動脈の性質上維持。無変化時skipは既に機能）
- update_mani系（系統A・手動ゲート）は今回の重さと無関係につき現状維持

### 実装順序の推奨
**A → C → B′ → D**（A/Cは無風で入る安全網＝鮮度を守る本丸。B′は鮮度そのまま・コミット/push集約。鮮度判断は 2026-07-21 に「毎時維持」で確定済み）
