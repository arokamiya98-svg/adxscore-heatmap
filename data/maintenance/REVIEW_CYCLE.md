# シグナル再評価サイクル設計 v1

> 作成: 2026-06-03 / 設計: メインおぱ（叩き台）
> 目的: ATR_WidthSignal シグナルとフィルター群を **データ資産として継続管理** する仕組み
> 対象: v4以降の全シグナル世代 / 全フィルター（現役・候補・引退すべて）

---

## 0. 設計思想

```
データ資産 = 一度作って終わらせない / 定期的に検証・更新する
            ↓
"動的な仕組み" がないと、フィルターは作った瞬間から腐る
```

### サイクルの三本柱

1. **定期検証**: 決まったタイミングで「まだ機能してるか」を測る
2. **判定ルール**: 機械的な引退・採用判定線
3. **記録の蓄積**: 各サイクルの結果を時系列で残す（後から振り返れる）

## 3. 引退・維持・追加の判定ルール（叩き台）

### 3.1 引退判定（フィルター単位）

「採用時の根拠」と「最新検証結果」を比較。**4段階の判定**:

| 状態 | 判定基準 | アクション |
|---|---|---|
| 🟢 **健在** | 最新PF ≤ 0.7（採用時と同水準） | 維持 |
| 🟡 **要観察** | 最新PF 0.7〜1.0 | 次回まで様子見、ログだけ取る |
| 🟠 **疑問符** | 最新PF 1.0〜1.3 | **物語が崩れていないかチェック**。崩れてたら引退候補 |
| 🔴 **引退候補** | 最新PF > 1.3 かつ N≥10 | **物語と数値が両方崩れた → 引退** |

### 3.2 物語チェック（数値より優先）

最新PFが多少悪くても、**構造的説明（物語）が依然として有効ならば維持**。
- 例: NONE × SELL がボツな理由 = 「D1方向不定期の売り」。この物語が崩れる = 「方向不定期の売りが機能する相場構造」が新規発生したことを意味する。**それは引退判断の前にあろさん独自の構造観察を入れる**。
- カイの数値報告だけで決めない（[[memory: bt-analysis-principles]] 「数値より物語が先」）。

### 3.3 新規候補追加ルール

| 段階 | 基準 |
|---|---|
| 候補入り | カイ分析で PF≤0.7, N≥10, 物語立つ |
| 仮採用 | おぱ+あろさんが「採用妥当」と判断 |
| 本採用 | 1サイクル（3ヶ月）フォワード稼働 → 仮採用時の物語が維持されてる |

### 3.4 閾値調整ルール（F7/F9のような可変閾値）

- 例: F7（DI_Spread_Tight）の閾値 1.0
- レビュー時に「閾値を 0.5 / 1.0 / 1.5 / 2.0 で試す」 → 最適値を更新
- 大幅変更（30%以上）は v5 として新規切り出し（v4は残す）

> `FILTER_REGISTRY.md` は各サイクルのレビュー結果を反映して**おぱが手動更新**。レビュー毎に最新版にする。

## 7. 想定されるリスク・限界

1. **DN局面サンプル不足**: 引退判定時もDN局面のサンプル充実が課題。3ヶ月後でも改善しないかも
2. **構造変化の早期検知**: 相場構造が変わった瞬間に気づける仕組みは無い（レビュー間の空白）
3. **判定ルールの硬直化**: 「PF<0.7 で健在」基準が時代遅れになる可能性 → 年次総点検でルール自体も見直す
