## 1. 札の性格づけ（設計思想）

- **D1環境札と同じ「環境認識」ファミリー**の一員。DXYは粗い札 — ADX(56)/DIは平滑で帯遷移に日単位かかる。速報性は要らない
- 本質は方向札ではなく**「迷い検出札」**: DXYが決まってるか迷ってるかを読む
  - 分析根拠: 死に帯は方向でなく「揺らぎ帯」（BUY×USD_DN揺らぎ PF0.62/33%、SELL×USD_UP揺らぎ PF0.77/29%、BT/FW両ソース一致）
  - 中間帯の曖昧さ系列の4例目（D1 ADX20-30罠 / ADX56 25-30凹み と同族）
- ラベル層の札。点数化しない。組合せ判断（シグナル方向×DXY状態）はあろさんの認知の仕事、札は状態表示まで

## 2. 深さラベル（実測分布ベース・段階化厳守）

|spread| = |DI+ − DI−|（DXY H1・ADX(56)付属DI・確定バー）

| ラベル | 閾値 | 滞在率* | 読み方（骨肉） |
|--------|------|--------|---------------|
| 拮抗 | <2 | 23% | **DXY読まない**。方向はコイントス、XAU内部だけで判断 |
| 揺らぎ | 2-5 | 29% | **⚠迷い帯**。特に「DXY追い風のはず」側が死ぬ（BUY時のドル安揺らぎ/SELL時のドル高揺らぎ） |
| 優勢 | 5-10 | 32% | 方向が意味を持つ。USD_UP優勢×BUY=押し目製造機（F後PF≈3）/ USD_DN優勢×BUYも可（両期間+） |
| 一方通行 | ≥10 | 17% | 情報はあるが**不安定帯**（BT/FW割れ）。過信しない・※印 |

- *滞在率はHFM USDIndex（MT5）実測。⚠️ **Twelve DataのDXYはバー構成が異なる**（24hインデックス vs 先物セッション）ため、導入時に閾値の妥当性検証が必要（§7-3）
- D1環境札の閾値（<5/5-10/10-16/≥16）とは**スケールが別物**。同じ意味論・別の物差し。UIで数値を混ぜない

## 3. データ経路（Twelve Data・widget側完結）

```
iPhone: d1_env_widget.js（既存widget内に追加）
  ├ 既存: d1_env.json fetch（GitHub Pages・XAU D1札）← 変更なし
  └ 追加: Twelve Data /time_series?symbol=DXY&interval=1h&outputsize=1500
       → JS内で Wilder DI+/DI-/ADX(56) を計算（atr_widget のWilder ATR実装と同流儀）
       → 最終確定バーの spread から深さラベル判定
       → 成功時: iCloud にキャッシュ保存 / 失敗時: キャッシュ表示＋「(キャッシュ)」注記
```

**設計判断の理由**:
- APIキーは既に端末のScriptable側にある（リポジトリ内はプレースホルダー＝公開リポジトリにキーを置けない）。Mac側で叩くと新規のキー管理が発生する
- Mac公開脚（hourly_sync→Pages）に依存しない = 2026-07-14のd1_env.json置き去り事故のような公開レグ障害でもDXY札は生きる
- VPS/MT5に一切触らない（USDIndex先物の限月ロール地雷も回避）
- Stage 9（Scriptable+Twelve）の確立済みスタックとエラー処理パターン（API Err→キャッシュ表示）をそのまま流用

## 4. widget内キャッシュ

- iCloud Scriptableディレクトリに `dxy_env_cache.json` を保存: `{fetched_at, adx56, di_plus, di_minus, spread, depth_label}`
- API失敗時はキャッシュを表示し、経過時間を小さく添える（atr_widgetと同パターン）
- キャッシュが24h超の場合は⚠表示（D1札の鮮度警告と同じ視覚言語）

## 5. UI表示（d1_env_widgetにDXY行を追加・D1札と同一視覚言語）

```
┌───────────────────────────┐
│ D1 環境        [⚠ n/n]     │
│ BEAR  ADX 31               │
│ DI -12.6   ⬊ 優勢          │
│ 5日: -13 〜 -8             │
│ ATR ↓収束 2日目            │
│ ─────────────────          │
│ DXY ⬆ 優勢 +9.8 (5d 8~13)  │ ← 追加行
└───────────────────────────┘
```

| 状態 | 表示 | 色 |
|------|------|----|
| 拮抗 | `DXY ─ 拮抗`（方向・数値なし） | グレー |
| 揺らぎ | `DXY 〰 迷い +3.2` | アンバー（警戒） |
| 優勢 | `DXY ⬆/⬇ 優勢 +9.8 (5d)` | DI基準の方向色（D1札と同じ） |
| 一方通行 | `DXY ⬆⬆/⬇⬇ 一方通行※ +12.4` | 方向色濃 + ※（過信しない） |

- 方向色はDI基準のみ。**良し悪しの色付けはしない**（USD_UP優勢×BUYが最強セルという逆説が根拠）
- 5dレンジ: Twelve時系列から直近5営業日の日次終値時点spreadで算出（優勢以上のみ表示）

## 7. 地雷・前提チェック

1. **Twelve DataでDXYが引けるか**（プラン制約の可能性）。NGなら代替: 主要ペア合成 or VPSルート再検討 — 判断持ち戻り
2. **APIコール予算**: free tierは8req/min・800req/day。既存atr/ratio系widgetと合算でリフレッシュ時にバーストしないか。DXYは1リフレッシュ1コール・粗い札なのでwidgetのrefresh間隔を長め（≥30分）に設定してよい
3. **閾値スケール検証**: 2/5/10はMT5(HFM USDIndex)実測由来。Twelve DXYで|spread|分布のp25/50/75を出し、段階の滞在率（23/29/32/17%）が概ね保たれるか確認。ズレが大きければ閾値を引き直す（意味論は維持）
4. **ADX(56)ウォームアップ**: Wilder再帰のため取得1500本中、先頭側~500本は収束用に捨てる（表示に使うのは終端のみなので実害なし・実装上の注意）
5. 週末バー: Twelveの土日欠落/薄いバーの扱い — 5dレンジは営業日ベースで数える（D1札の土曜行地雷と同根の意識）

## 8. v1で見送るもの（理由つき）

- **ADX56帯（25-30凹み）の表示**: N=38の弱シグナル段階。札に刻むには早い（フォワードで追認後）
- **順風/向かい風の自動判定表示**: 組合せ判断はあろさんの認知の仕事。札は状態まで
- **ratio_widget への重複表示**: 認知負荷（足すより絞る）。環境札はd1_env_widgetに集約
- **VPS EAルート**（旧§3案）: Twelve確定により廃案。ただし§7-1がNGの場合の控えとして記録を残す
