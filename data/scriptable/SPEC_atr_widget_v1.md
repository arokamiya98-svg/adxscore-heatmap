## 1. 設計コンセプト

```
Scriptable = 「価格を見ずに認識するためのライブツール」
  ・チャート画面を開かなくても状態がわかる
  ・数値ラベルだけで「今やる/やらない」が決まる
  ・Wilder ADX依存を捨てる = 精度議論を回避
  ・あろさん感覚を CONFIG 値として固定 = 認識の一貫性
  ・閾値タップで「シンプル編集」可能（コードに触らなくていい）
```

### Widget Web (heatmap_v14) との住み分け

| | Widget Web | Scriptable |
|---|---|---|
| 視点 | マクロ・俯瞰 | ライブ・顕微鏡 |
| 更新 | 1日1回 | 15分（iOS WidgetKit 最短） |
| 時間軸 | 週次〜D1〜H4 | 形成中 H1 / H4 |
| 役割 | 「今週どんな局面か」 | 「今この瞬間どう動いてるか」 |
| データ源 | MT5 → CSV → HTML | Twelve Data API |
| 配置 | ホーム画面 上半分 | ホーム画面 下半分 |

## 4. 状態色

### 4.1 ATR系（HIGH 1本のみ、2段階）
| 状態 | 色 | 意味 |
|---|---|---|
| ATR < HIGH | 🟢 緑 | 適正帯（まだ余裕） |
| ATR ≥ HIGH | 🔴 赤 | 加熱（注意） |

> **凪判定（LOW）は意図的に外した**。
> 理由: レンジで加熱は定義できるが、BU 来たらメンタルブロックになる懸念あり。凪はあろさん側で判断（あろさん明示の方針 2026-06-05）。

### 4.2 トレード回数（3段階）
| 状態 | 色 | 意味 |
|---|---|---|
| count < ideal | 🟢 緑 | 余裕 |
| ideal ≤ count < limit | 🟡 黄 | 適正範囲だが注意ライン突入 |
| count ≥ limit | 🔴 赤 | 超過、要振り返り |

> **設計意図**: `ideal=3` は「ここまでに仕留めたい目標」、`limit=5` は「これ以上は明らかにやりすぎ」。
> 「上限5回やるぞ！」ではなく「3回で済ませるのが理想」のイメージ。

## 5. 設定値（CONFIG）

### 5.1 コード冒頭 CONFIG（コードに直書き、初期値兼デフォルト）

```javascript
const CONFIG = {
  // Twelve Data API
  api_key: "<TWELVE_DATA_API_KEY>",
  symbol: "XAU/USD",

  // ATR 加熱境界（HIGH のみ、1本ライン）
  // ※これらは設定ダイアログでも編集可能。編集後は config.json が優先。
  h1_atr16_high:  13.3,
  h1_atr32_high:  13.3,
  h4_atr8_high:   30.0,

  // 今週トレード回数（コード固定、ダイアログ編集対象外）
  weekly_ideal:   3,     // 余裕→注意の境界
  weekly_limit:   5,     // 注意→超過の境界
  reset_weekday:  1,     // 0=日, 1=月... 月曜00:00 にリセット

  // 表示

};
```

## 6. データ取得

### 6.1 エンドポイント
- `GET https://api.twelvedata.com/time_series`
- パラメータ: `symbol=XAU/USD&interval=<1h|4h>&outputsize=<N>&apikey=<KEY>`

### 6.2 必要なリクエスト
| Call | interval | outputsize | 用途 |
|---|---|---|---|
| 1 | `1h` | **64** | H1 ATR(16) と ATR(32) 計算（ATR(32) 初期化のため 64本） |
| 2 | `4h` | 24 | H4 ATR(8) 計算 |

→ **1更新あたり 2 call**
→ 15分更新 × 96回/日 = **192 call/日**（無料枠 800 req/day に余裕）

### 6.3 レスポンス例
```json
{
  "values": [
    { "datetime": "2026-06-05 07:00:00", "open": "4477.00", "high": "4480.50", "low": "4475.10", "close": "4478.20" },
    ...
  ]
}
```
※ `values` は新しい順で返る。ATR計算では古い順に並べ替えてから使う。

## 7. ATR 計算（Wilder方式）

### 7.1 True Range
```
TR_i = max(
  high_i - low_i,
  |high_i - close_{i-1}|,
  |low_i - close_{i-1}|
)
```

### 7.2 ATR
- 初期 ATR(N) = TR の単純平均（最初の N 本）
- 更新式（Wilder smoothing）:
  ```
  ATR(N)_i = (ATR(N)_{i-1} × (N - 1) + TR_i) / N
  ```

### 7.3 MT5 iATR との整合性
- Wilder式 + 初期=単純平均 で MT5 iATR と数値一致を目指す
- 丸め誤差・初期化バー数の扱いで微小ズレの可能性あり
- **実用上の許容**: 状態色判定（HIGH 境界の上か下か）が一致すれば OK
- フォワード比較: 実装後 1週間ほど MT5 iATR と Scriptable 値を比較ログ取って許容差確認（あろさん運用）

## 8. インタラクション仕様

### 8.1 「+1」ボタン（カウントアップ）
- タップで `?action=increment`
- 起動 → カウンタ +1 → ファイル保存 → 即再描画

### 8.2 「⟲」ボタン（リセット）
- タップで `?action=reset`
- 起動 → カウンタ 0 / last_reset 更新 → 保存 → 再描画
- 確認ダイアログ無し（押し間違い防止はアイコンを小さく配置して物理的に対策）
- iOS の長押し検出はウィジェット内では制限があるため別タップ領域で実装

### 8.3 閾値タップ → 設定ダイアログ
- タップで `?action=settings`
- Scriptable アプリが起動して `Alert` を表示:
  ```
  ┌─────────────────────────┐
  │  ATR 閾値設定            │
  ├─────────────────────────┤
  │  H1 ATR(16) HIGH        │
  │  [   13.3    ]           │
  │                          │
  │  H1 ATR(32) HIGH        │
  │  [   13.3    ]           │
  │                          │
  │  H4 ATR(8) HIGH         │
  │  [   30.0    ]           │
  ├─────────────────────────┤
  │   キャンセル    保存     │
  └─────────────────────────┘
  ```
- 「保存」タップ → `config.json` 書き込み → Script.complete()
- 次回ウィジェット再描画時に新しい閾値が反映

### 8.4 自動リセット判定
- スクリプト起動時、CONFIG.reset_weekday の **直近の該当日** を計算
- `last_reset` < 計算結果 なら、`week_count = 0` にリセット
- 例: 今日 2026-06-09 (火), last_reset 2026-06-02 (月), reset_weekday=1 (月)
  → 直近月曜 = 2026-06-08 → last_reset < 2026-06-08 → リセット実行

### 8.5 ATR下抜け通知（2026-06-22 実装）

**目的**: ATRが落ち着き（ボトムアウト寄り）へ入った変わり目を逃さない。加熱通知ではなく**落ち着き側**の検知 ＝ あろさんの「ATRボトムアウト狙い」の構えどき合図。加熱通知（チャートへの引き寄せ懸念）とは逆ベクトルで、認識ツール思想と整合する。

| 項目 | 仕様 |
|---|---|
| トリガー | 各ラインが 🔴加熱(≥閾値) → 🟢落ち着き(<閾値) に切り替わった瞬間（下抜けエッジ） |
| 対象 | H1 ATR(16) / H1 ATR(32) / H4 ATR(8) を**各ライン独立**に判定 |
| 発火回数 | 下抜けの瞬間に1回。🟢継続中は黙る |
| 再発火 | 一度🔴へ戻り、再び🟢へ下抜けたら再発火 |
| まとめ | 同一更新で複数本が同時に下抜けたら1通知に列挙（連発防止） |
| 通知文 | タイトル `💧 落ち着き下抜け: H1(16)` ／ 本文に各ラインの現値・閾値 |
| 継続確認 | 通知は変わり目のみ。今も🟢かは**ウィジェットで見る**（役割分担） |
| 判定タイミング | 自動更新／背景タップ時のみ（+1・リセット・設定変更では鳴らさない） |
| 粒度 | iOS のウィジェット更新依存（15分〜不定）。落ち着きは継続するため実用上問題なし |

**実装メモ**:
- `state.json` に `prev_status`（各ラインの前回赤緑）を保持してエッジ検出
- `prev_status` は有効値（N/A以外）のときだけ更新 → API一時失敗を跨いでも遷移を取りこぼさない
- settings で閾値変更した直後は判定スキップ ＝ 基準を取り直し、見かけ上の遷移で誤発火させない
- ⚠️ **実機確認事項**: ウィジェット文脈から `Notification` が飛ぶか。飛ばない場合はショートカットのバックグラウンド実行へ切替（保険）

## 9. 永続化（iCloud Drive）

```
iCloud Drive/Scriptable/
├── atr_widget_state.json    ← トレード回数
└── atr_widget_config.json   ← ATR 閾値（タップ編集対象）
```

### 9.1 atr_widget_state.json
```json
{
  "week_count": 3,
  "last_reset": "2026-06-02",
  "prev_status": { "h1_atr16": "green", "h1_atr32": "red", "h4_atr8": "green" }
}
```
> `prev_status` は下抜け通知のエッジ検出用（§8.5）。各ラインの前回の赤緑状態を保持。

### 9.2 atr_widget_config.json
```json
{
  "h1_atr16_high": 13.3,
  "h1_atr32_high": 13.3,
  "h4_atr8_high":  30.0
}
```

### 9.3 読み込み優先順
1. `atr_widget_config.json` が存在すればその値を使う
2. 存在しなければ CONFIG（コード冒頭）の初期値を使う
3. 不正値（NaN、負数）は無視して初期値にフォールバック

### 9.4 iCloud 同期
- iPhone / iPad 間で自動同期
- 同時編集競合は考慮しない（ユーザー1人運用）

## 10. エラー処理

| ケース | 挙動 |
|---|---|
| API 応答失敗（4xx/5xx） | 直近の値を表示 + 「API Err」ラベル |
| ネットワークなし | 直近の値 + 「Offline」ラベル |
| 初回起動でデータなし | 「Loading…」表示 |
| iCloud File 読み込み失敗 | 初期値で新規作成 |
| ATR 計算で 0除算 | 該当行を「N/A」表示 |
| 設定ダイアログでの不正入力 | 「無効な値」alert を出して保存中止 |
