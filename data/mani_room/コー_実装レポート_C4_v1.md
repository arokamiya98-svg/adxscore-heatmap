### 1.2 関数構造の再設計 (重要)

#### 既存実装の問題発見
v1.2 では `TraceMaeMfeH1_Segmented(...)` を呼び出していたが、**関数本体の実装が存在しない** (ファイル内に定義なし) ことを確認。これはコンパイル時にエラーとなり、v1.2 は実際には MT5 で動かなかった可能性がある。

→ 静的レビュー段階で発見。改修と同時に修正。

#### 汎用関数への統一
旧 `TraceMaeMfe48h` を時間別セグメント対応の汎用関数 `TraceMaeMfe_Segmented` に置換。1関数定義で全TF (H1/H4/D1) に対応。

```cpp
bool TraceMaeMfe_Segmented(
    ENUM_TIMEFRAMES tf, int entry_shift, int trace_n,
    double entry_price, const string &direction,
    int seg1, int seg2, int seg3, int seg_full,   // ★ セグメント本数を引数化
    double &mfe_s1, double &mae_s1,                // 各セグメントの MFE/MAE
    double &mfe_s2, double &mae_s2,
    double &mfe_s3, double &mae_s3,
    double &mfe_sf, double &mae_sf,                // seg_full = 全体 (48h相当)
    int &mfe_bar_idx_full, int &mae_bar_idx_full,  // bar_idx は seg_full のみ
    int &bars_traced
);
```

#### 各TFの呼び出し設定
| TF | trace_n | seg1 | seg2 | seg3 | seg_full | 意味 |
|---|---|---|---|---|---|---|
| H1 | 48 | 12 | 24 | 36 | 48 | 12h/24h/36h/48h |
| H4 | 12 |  3 |  6 |  9 | 12 | 12h/24h/36h/48h |
| D1 |  2 |  1 |  1 |  2 |  2 | 24h/24h/48h/48h (12h/36h は D1 解像度なし) |

D1 の seg1/seg3 は CSV 非出力なので dummy 変数で受ける。

### 1.5 既存 48h ロジックとの整合性

#### 計算式の同一性
旧 `TraceMaeMfe48h` と新 `TraceMaeMfe_Segmented` の `seg_full` 部分は、計算式・走査範囲・bar_idx 算出方法が完全に同一。
- 走査範囲: `start_shift = entry_shift - 1` 〜 `end_shift = max(0, entry_shift - trace_n)`
- BUY: `favor = high - entry`, `adverse = entry - low`
- SELL: `favor = entry - low`, `adverse = high - entry`
- bar_idx 変換: `n_bars - series_index_k` (entry 後 1..N)

→ **既存 48h 値は変化しない** (リグレッションなし)

#### セグメント値の累積定義
`seg1 ⊆ seg2 ⊆ seg3 ⊆ seg_full` の累積関係:
- `bar_no = n_bars - k` (entry 後の何本目)
- `if (bar_no <= seg_N)` で各セグメントに条件加算
- 結果: 値が `seg1 ≤ seg2 ≤ seg3 ≤ seg_full` の順に単調増加する

これは指示書 5.1 の「N時間以内の累積最大」要件と一致。

### 1.6 部分追跡(直近トレード等)の取り扱い
- entry_shift < trace_n の場合、`n_bars = entry_shift` で走査
- 各セグメントは「実取得バー本数 ∩ seg N」の範囲で計算
- 例: 36h までしか取得できなかった H1 トレードは
  - 12h/24h/36h: 完全値
  - 48h: 「実取得 36本まで」の値 (partial)
- `bars_traced` 列で実走査本数を保持し、partial 判定可能

- **案A: C4-B 命名を C4-A 流に合わせて統一**

- **案B: 既存維持 (現状)**
  - 利点: 後方互換完全維持、追加実装不要
  - 欠点: マニ第2弾 UI で C4-A と C4-B を読むときに微妙に違う命名を意識する必要

→ **本レポート時点ではどちらも実装可能**。マニ実装者の好みとあろさんの判断に委ねる。コーは判断しない。
