### U3: drilldown-table の H1 ATR / H4 ATR バグ修正

**バグ原因**:
- `_enr_by_key` 構築時、enriched CSV のキー名を誤参照
- 旧: `_er.get("trade_date")` ← **このキーは存在しない**（CSV 実列は日本語「約定日」）
- 結果: `_td` が常に空文字列 → `(_td, _dir)` の date 側が空 → マッチゼロ → `_h1_atr_abs` / `_h4_atr_abs` が全件 None → JS 側で「—」表示

**修正内容** (`scripts/generate_daily_calendar.py` 2232-2245 行付近):
```python
_enr_by_key = {}
for _er in enriched_map.values():
    try:
        # 「約定日」は "2026/03/17 13:00" 形式 → 先頭10文字を "YYYY-MM-DD" に変換
        _src = (_er.get("約定日") or "").strip()
        _td = _src[:10].replace("/", "-") if _src else ""
        _dir = (_er.get("direction") or "").strip().upper()
        if _td and _dir:
            _enr_by_key.setdefault((_td, _dir), []).append(_er)
    except Exception:
        pass
```

**修正確認**:
- 修正前: `"h1_atr_abs": null, "h4_atr_abs": null` (全件 None)
- 修正後: `"h1_atr_abs": 24.8322, "h4_atr_abs": 53.2137` 等、全 26 件で実値表示
- `grep -c '"h1_atr_abs": null'` → 0 (null ゼロ確認)
- `grep -c '"h4_atr_abs": null'` → 0 (null ゼロ確認)

**副次効果**:
- H1/H4/D1 の DI 方向 (`h1_dir`, `h4_dir`, `d1_dir`) も全件取得できるようになった
- 詳細分析タブの「方向整合」フィルタ（3揃/2揃/不揃）が正しく動く（旧は全て "不揃" に偏倒していた可能性）

### U4: 全数値列の昇降ソート対応

**ソート対応列**（drilldown-table / detail-table 両方）:
- 数値列: `pl` (損益) / `mfe` / `mae` / `h1_atr_abs` / `h4_atr_abs` / `h1_atr_ratio`
- 文字列列: `date` / `order` / `pattern` / `d1_pattern` / `star`

**実装方針**:
- 数値列は `NUMERIC_DRILL_KEYS` / `NUMERIC_DETAIL_KEYS` Set で識別
- `Number(va) - Number(vb)` で数値ソート（文字列ソートにならない）
- **None は常に末尾固定**（昇降どちらでも）
  - `aNil = (va == null || (isNumeric && isNaN(va)))` で None・NaN を一括判定
  - aNil なら 1、bNil なら -1 を返してソート末尾へ
- ヘッダ初回クリック: 昇順 ▲ / 同列再クリック: 降順 ▼ / 別列クリック: その列の昇順

**探索目的（あろさん原文）**:
> H1 ATR 昇順ソート 7.1, 7.8, 8.3, 8.5, 9.1, ..., 18.4 で並べたい。
> 目的: どの ATR 帯から勝率や MFE が変化するか探索するため。

→ 実装後、たとえば「H1 ATR」列をクリックすると `23.6875 → 24.8322 → 38.2634 → ... → 75.835` と並ぶ。MFE / 勝敗 / 反省タグ列を眺めながら「どの ATR 帯から景色が変わるか」を直接観察可能。

**detail-table 側も同様に強化**:
- 旧実装はソート機能あったが None 末尾固定が未実装 → `null` が先頭に来てしまっていた可能性
- 同じ `NUMERIC_DETAIL_KEYS` ロジックを適用
- 旧 `.detail-table thead th` セレクタが drill-table も拾ってしまうバグの懸念があったので、`.detail-table:not(.drill-table)` で詳細分析タブ専用に限定

## 3. マニ判断で進めた詳細設計

### 3.1 SVG 扇形と凡例の両方をクリック対象にした理由

- スマホ・iPad での扇形タップは小さい扇形だとミスタップしやすい
- 凡例行は十分な高さがあり、確実にタップできる
- どちらをタップしても同じ `data-pie-key` / `data-pie-value` を持つので、JS 側は同一ハンドラで処理
- 見た目の一貫性: 凡例にも `pie-slice-active` クラスでハイライト

### 3.2 「全件表示」ボタンを drilldown ヒント行に配置

- 別ボタン領域を作ると視線が散る
- ヒント文「列ヘッダクリックで昇降ソート」と並べることで、操作系の集約感を出す
- スタイルは控えめ（背景 `#0e1a2a`、border `#1a3454`）

### 3.3 初期表示を「非表示」→「全件表示」に変更

- 旧: 何か操作するまで drilldown-wrap が `display:none`
- 新: 初期から全 26 件を表示
- 利点: 「最初に何を見ればいいか」が明確、フィルタ概念の学習コスト低減
- フィルタ未操作時の見出しは「全件表示 (26件)」で誤解を防ぐ

### 3.4 d1_phase / won_label / dow_label を TRADES に追加

- 円グラフの凡例値（「BU」「勝ち」「月」など）とトレードの属性を JS 側で `===` 比較できるよう、表示語そのままを TRADES オブジェクトに格納
- `d1_phase` は `phase_counts` と同一ロジック（d1_adx<18=RANGE → d1_pattern in (BU,PD) → else その他）で算出
- 内部キー (`pattern`, `won`, `d1_phase`, `dow`) は番人観点の「内部識別子」であり、UI には登場しない

## 5. 番人レビュー観点への先回り回答

### Q: パターン名がコード/UI に登場していないか
- A: TRADES オブジェクトの `pattern` フィールド名は「反省タグ列」と同一語源で、あろさん意思決定タグの内部キー（PAT-A/B/C/D 等）。BT 由来のパターン名 (RISING_DECEL/EXPANDING/CONTRACTING) は **0 件**（`grep -ci` で確認）。

### Q: 円グラフ扇形に判定ラベルが付いてないか
- A: `<title>` ツールチップは「<カテゴリ名>: <件数>件 (<%>%)」のみ。「最強」「優位」「危険」等の判定語なし。凡例も同様。

### Q: ソート結果に「並び順=優先度」と読める表現がないか
- A: ヒント文は「列ヘッダ（…）クリックで昇降ソート」のみ。「優先」「ランキング」「順位」「優位」等の語句なし。
- ▲▼ アイコンは方向表示のみ（`#4a90e2` 青）、優劣の暗示なし。

### Q: ATR 絶対値が正しく enriched から流れているか
- A: U3 修正により全 26 件で実値表示。サンプル: `h1_atr_abs: 24.8322`, `h4_atr_abs: 53.2137`。null ゼロ。

### Q: 「凪/弱/中/HOT」等の評価語が新規追加されていないか
- A: 新規追加 **なし**。既存 v1.3 以前の env-memo 表示「凪」は今回触らず（指示書範囲外）。
