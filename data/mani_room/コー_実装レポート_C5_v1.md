## 0. 結論 (TL;DR)

- **v1.31 までスナップショット (ATR/ADX/DI/Pattern/Cross/Phase) に未来情報が混入していた**
- 原因: `iBarShift(exact=false)` が返すのは「指定時刻が**属する**バー」の shift であり、その shift で取得した Indicator buffer 値は**バー終値時点 (= エントリーから最大 1 バー後) の値**だった
- 修正: スナップショット取得用 shift を `+1`（直前確定バー = エントリー時点で利用可能だった確定値）に補正
- **CSV ヘッダ / 列数 / 順序: 完全不変 (72列)**
- **TraceMaeMfe_Segmented の挙動も不変** (引数にエントリーバー shift を渡す形を維持)
- **bar_time メタデータも不変** (エントリーバー時刻を記録)
- version: `1.31` → `1.32`

## 1. 問題の根本原因

### 1.1 `iBarShift(symbol, period, time, exact=false)` の真の挙動

MQL5 公式ドキュメントベース:

```
iBarShift(time, exact=false)
  → 指定 time を「含む」バーの shift を返す
  → 例: H1 で指定時刻 13:30 → 13:00 バー (open_time=13:00, close=14:00) の shift
```

「直前確定バー」ではない。あろさん指摘の通り、コメントに「直前確定バー」と書いていたのは**誤解を招く記述**だった (v1.31 までの実装も誤解と整合してしまっていた)。

### 1.2 MT5 Indicator buffer の時間整合性

MT5 の Indicator buffer (`iATR`, `iADX` 等) は、各バーで **クローズ時点 (= バー終値) の値が確定**する。
- バー [13:00-14:00] の `iATR[shift]` の値 = **14:00 時点の ATR 値**
- これはバー開始時点 (13:00) ではなく、バー終了時点で初めて確定する

### 1.3 二つの組み合わせで生じた未来情報混入

```
エントリー時刻: 13:30 BUY
  ↓ iBarShift(exact=false)
sh_h1 = H1 バー [13:00-14:00] の shift
  ↓ GetBufValue(hATR, 0, sh_h1)
ATR 値 = 14:00 時点の値 (= エントリーから 30 分後の値)

→ エントリー時点 (13:30) では未確定 = 未来情報混入
```

最悪ケース:
- H1: 最大 60 分後
- H4: 最大 4 時間後
- D1: 最大 1 日後 の値で「エントリー時点の市場環境」を記録してしまっていた

### 1.4 研究目的との照合

研究目的 (絶対固定 / `feedback_research-purpose-and-rules.md`):
> 「日時+価格をキーとして、エントリー時点の市場環境を後付け取得」

v1.31 までの実装は、エントリー時点で**知り得ない**未来情報を含んでいた。「後付け取得」の意味から逸脱しており、**研究目的違反**。

## 2. 修正方針

### 2.1 戦略: shift +1 (= 直前確定バー)

```cpp
// v1.31 (誤)
int sh_h1 = iBarShift(sym, PERIOD_H1, entry_server, false);
double atr = GetBufValue(hATR, 0, sh_h1);  // バー終値時点 = 未来

// v1.32 (正)
int sh_h1_bar = iBarShift(sym, PERIOD_H1, entry_server, false);  // メタデータ用
int sh_h1     = sh_h1_bar + 1;                                    // スナップショット用 = 直前確定バー
double atr    = GetBufValue(hATR, 0, sh_h1);                      // エントリー時点で既に確定済みの値
```

あろさん解釈と整合:
> ATRの取得位置がおそらく終値かEXIT基準かな？
> いいね！それで始まり値と同じになる
> (= shift+1 で直前確定バー = エントリーバーの始値時点の値)

### 2.2 関数引数の用途分離

| 用途 | 渡す shift | 理由 |
|---|---|---|
| `GetBufValue` (ATR/ADX/DI) | `sh_h1` (= +1) | バー終値 = 直前確定バー終値 = エントリー時点既知 |
| `CalcAtrMedian` | `sh_h1` (= +1) | 中央値ウィンドウの起点を直前確定バーに合わせる |
| `ComputeH1Pattern` | `sh_h1` (= +1) | 関数内で同様に CopyBuffer する、起点を直前確定バーに |
| `FindCrossFromHandles` | `sh_h4` / `sh_d1` (= +1) | クロス検索開始点を直前確定バーに |
| `TraceMaeMfe_Segmented` | `sh_h1_bar` / `sh_h4_bar` / `sh_d1_bar` (= 元のエントリーバー) | 関数内で `entry_shift - 1` 起点で追跡するため、エントリーバーを渡して仕様維持 |
| `iTime` (bar_time メタデータ) | `sh_h1_bar` / `sh_h4_bar` / `sh_d1_bar` | エントリーが属するバーの時刻を記録 (CSV メタデータ列の意味は不変) |

## 8. 残課題 / 補足

### 8.1 もし「エントリー時刻=バー開始ぴったり」のレアケース

`iBarShift(exact=false)` の挙動上、`open_time == entry_server` のケースでもそのバーが返る。
このとき `sh_h1_bar + 1` で直前確定バー shift になるため、ロジック的にも問題なし。

### 8.2 もし `sh_h1_bar = 0` (= 現在形成中バーがエントリーバー)

`sh_h1 = 0 + 1 = 1` となり、直前確定バー shift になる。
MT5 がスクリプト実行時に過去日付の trade を処理する場合は通常そんなことはない (常に過去バー) ので、想定外動作はなし。

### 8.3 v3 BT / FW ログとの整合

v3 系の他 mq5 (`ATR_WidthSignal_v4.mq5` 等) のリアルタイムシグナル発火ロジックは、`shift=1` (直前確定バー) を見ているはず → 整合性的にも `Trade_Snapshot_Builder` を直前確定バーに合わせるのが本筋。
