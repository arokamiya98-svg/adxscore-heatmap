## 1. エラーの根本原因

### 1.1 MQL5 言語仕様
**MQL5 関数引数の最大数: 64個**
- 公式の硬い制約 (MetaTrader 5 リファレンス: ユーザ定義関数)
- 65 番目以降の引数は内部的に「打ち切り」される
- パーサーは 65 番目以降の `type name` を「同名再定義」と誤検出 → `redefinition of parameter` エラー

### 1.2 v1.3 で何が起きたか
- v1.2 までの `WriteRow()` は 64引数 (限界ギリギリ)
- v1.3 改修で H4 6列 + D1 2列 = 8引数 + α (実態は 9引数増) 追加 → 73引数
- 64 を超えた瞬間に下記エラー連鎖:
  ```
  too many parameters                       (line 1097)
  redefinition of parameter 'h1_mae_36'     (line 1097)
  redefinition of parameter 'h4_mfe_12'..'d1_mae_24'  (line 1098-1101)
  wrong parameters count, 73 passed, but 64 requires  (line 534, 呼び出し側)
  undeclared identifier 'h1_mfe_36'..'d1_mae_24'  (line 1199-1224, 関数本体内)
  ```
- 計 21 errors

### 1.3 v1.2 がなぜ通っていたか
- v1.2 時点の WriteRow 引数数を数え直し: 64引数で限界ギリギリ
- 1引数でも追加した瞬間に破綻する状態だった

---

## 2. 修正方針: 構造体化

### 2.1 戦略
- 73フィールドを `TradeSnapshotRow` 構造体に集約
- WriteRow を `void WriteRow(int fh, const TradeSnapshotRow &row)` に変更 (2引数)
- 呼び出し側は構造体に値を詰めてから渡す

### 2.2 なぜ構造体が最適か
| 案 | メリット | デメリット |
|---|---|---|
| **A. 構造体化 (採用)** | 1関数で完結、可読性高、フィールド名で追跡可 | コード量増 (詰め込み処理) |
| B. WriteRow 分割 (前半/後半) | 引数数削減 | ヘッダ整合維持コスト、CSV 1行を 2 関数で書く設計負債 |
| C. グローバル変数経由 | 引数 0 | スレッドセーフ性なし、副作用大、追跡困難 |

→ **A 案を採用** (MQL5 で C++ struct と同等の使い方が可能、参照渡しでコピーコストもゼロ)

## 4. WriteRow 改修箇所

### 4.1 シグネチャ変更 (1251行)

**Before** (v1.30):
```cpp
void WriteRow(int fh,
              const string &trade_id, const string &entry_jst, ...
              // 72引数の羅列 (合計73引数で 64 超過)
              double d1_mfe_24, double d1_mae_24)
```

**After** (v1.31):
```cpp
void WriteRow(int fh, const TradeSnapshotRow &row)
```

### 4.2 関数本体: 全フィールド参照を `row.field_name` 形式へ
- 例: `trade_id` → `row.trade_id`
- 例: `h1_mfe_usd` → `row.h1_mfe_usd`
- 計算ロジック・出力順序・条件分岐 (`if(row.h1_ok)` 等) は完全不変

### 4.3 副次バグ修正: d1_ok=false 時の末尾カンマ
**Before** (v1.30) [71列出力バグ]:
```cpp
if(d1_ok) {
   line += DoubleToString(d1_mfe_24, 3) + ",";
   line += DoubleToString(d1_mae_24, 3);    // 2フィールド分
} else {
   line += ",";                              // ← カンマ1個 = 1フィールドの空+区切り のみ。
                                             //    [71]空 + [72] が存在しない → 71列で行終端
}
```

**After** (v1.31) [72列正しく]:
```cpp
if(row.d1_ok) {
   line += DoubleToString(row.d1_mfe_24, 3) + ",";
   line += DoubleToString(row.d1_mae_24, 3);
} else {
   line += ",";   // [71]空 + 区切り + [72]空 (末尾フィールドなので末尾カンマなし)
}
```

→ 実は元の `","` でも、71列目「空」、区切り「,」、72列目「空」の解釈で 72列分のフィールドが構成される (CSV的に `,,foo` は 3フィールドと等価で末尾 `,,` も 3フィールド)。
→ **追加修正不要** (CSV パーサ的に同等)。コメントで意図を明示するのみ。
