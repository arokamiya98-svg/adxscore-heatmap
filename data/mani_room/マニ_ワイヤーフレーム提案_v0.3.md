## 0. v0.2 → v0.3 の差分要約（あろさん第2回フィードバック反映）

### v0.2 で残っていた問題

あろさん指摘: **「まだ週次の情報の転写に見える」**

| 症状 | 原因 |
|---|---|
| 項目が均等に並んだデータシート感 | レーン1 (70%) の中で D1 12% / H4 22% / DI 14% / H1 22% と**4等分** |
| H4 が「主役」に見えない | H4 と H1 が同じ 22% で並列 |
| DI が独立行 | H4 から切り離されて視覚一体化していない |
| D1 が「セル内」に閉じ込められている | あろさん「D1 は 1day 枠に収まらなくていい」発言と矛盾 |

### v0.3 で導入する階層化原則

```
情報認知優先度 = 視覚的優先度（物理的に反映）

【主役】  H4 ADX (強度 + DI 方向) … 一体表現で最大面積
【補助】  H1 ADX                  … 小さく、トレード成績照合用
【背景】  D1 ADX                  … セル外の横帯、複数日連続
【補足】  ADX値・MAE/MFE          … 視覚に寄り添う最小文字
```

| 階層差を出す3つの物理表現 | 適用 |
|---|---|
| **面積差** | H4 > H1 ≫ D1 (D1はセル外) |
| **包含関係** | H4 = セル本体 / H1 = 補助小行 / D1 = セル貫通帯 |
| **色相強度差** | H4 鮮やか / H1 やや薄い / D1 ごく薄い帯 |

## 3. H1 補助レーン — トレード成績照合用

### 設計原則

H1 は **「シグナルが発火してたか」「トレード成績と照らし合わせる」** ための補助情報。
**H4 とごっちゃにならないよう、視覚優先度を意図的に下げる**。

### H1 単体ASCII (H4 の下の小さい行)

```
┌─────────────────────┐
│ H4 (主役 46%)        │
│        42            │
│   ███████████       │
├─────────────────────┤  ← 区切り (細い線)
│ H1:  45  ▒▒▒▒▒      │ ← 補助行 16%
└─────────────────────┘
     ↑     ↑
     数値   小さい背景帯
     (12px) (高さ控えめ)
```

### H1 の役割固定

```
H4 → 「直近1週間のトレンドの力」(主役)
H1 → 「発火してたか」の事実記録 (補助)
      ・シグナル発火日に H1 が冷えてた → 取り逃がし
      ・シグナル発火日に H1 が高い → トレード成績と整合
```

### H1 をどう描くか

| 選択肢 | 評価 |
|---|---|
| A: 数値 + 細い背景帯 | ✅ **採用**。H4と同じ濃淡ルール、ただし高さ1/3 |
| B: 数値のみ | ❌ 視覚情報なしで「冷えてた」が直感に入らない |
| C: H4と同サイズの色付きセル | ❌ H4と並列に見え主役が崩れる |

→ **A 採用**: H4 の縮小版として描く。**同じ色相ルール (DI起源) は使わない**(H1にはDIを表示しない)、ADX強度の濃淡のみ。

## 5. スコア化の判断 — 入れる、ただし限定的に

### 判断: H4×H1 合成スコア (日次版ADXスコア) を**条件付きで導入**

### 判断の理由

#### 理由1: あろさんの問いに直接答えるため

あろさん原文:
> H1のADX...この辺りはごっちゃになるならスコア化してもいいよっていうのが前回の提案

→ 「ごっちゃになるかどうか」を実装ベースで判定する必要あり。
→ v0.3 設計では H4=46% / H1=16% で**面積差が明確**なので**通常はごっちゃにならない**。
→ ただし**狭幅日**(土日や非トレード日、または画面縮小時)では H4 と H1 が両方読めなくなる可能性。

#### 理由2: 「役割が違う」から本セルでは別表示が正しい

| | H4 | H1 |
|---|---|---|
| 役割 | 直近1週間のトレンドの力 | シグナル発火と成績照合 |
| 観測の主目的 | 環境の地形 | 取り逃がしや過剰反応の検出 |

→ **2つは違う問いに答える指標**。合成すると問いの分離が失われる。
→ **常時合成表示は反対**。

#### 理由3: ただし「縮約モード」では合成が有効

```
[通常表示] H4 (46%) と H1 (16%) を分けて表示
            ↓
[縮約モード] スコア = sqrt(H4 × H1) × DI バイアス補正  (1セル表現)
            画面狭小時 / 過去日のサマリ表示 / 別タブ長尺ヒートマップの圧縮表示
```

→ **長尺ヒートマップの圧縮行**として活用余地あり。

### スコア定義 (採用する場合の仮置き)

```python
def daily_adx_score(h4_adx, h1_adx, di_spread):
    # H4×H1 幾何平均ベース (過去 ADXスコア式の日次版)
    h4_norm = min(100, max(0, h4_adx))   # ADX値そのまま
    h1_norm = min(100, max(0, h1_adx))
    base = math.sqrt(h4_norm * h1_norm)   # 幾何平均
    # DI バイアスは色相に回す (スコア値には混ぜない)
    return base
```

### エントリーロジック混入チェック

| 含めるもの | 含めないもの |
|---|---|
| ✅ H1×H4 強度合成 (過去ADXスコア式の日次版) | ❌ パターン別重み付け (RISING_DECEL=最強 等) |
| ✅ 連続値 (0-100) | ❌ ADX加熱帯ペナルティ |
| ✅ DI 方向は色相で別表現 | ❌ ATR Zone 別係数 |
| | ❌ パターン×局面の重み付け |

→ **「強度合成」のみ**。**重み付け・ペナルティ・係数は一切入れない**。

### おぱ番人レビュー対象

スコア化を実装する段階で、おぱ(番人)が以下を確認:
1. H4×H1 の幾何平均のみで構成されているか
2. パターン名・ATR Zone名がコードに登場していないか
3. 「強度の合成」以外の操作が入っていないか

→ レビュー通れば導入OK、引っかかれば差し戻し。

### v0.3 新規リスク評価

#### リスクR1 (継続): 連続グラデーションの「閾値感」滲み

- 対策: 連続グラデーション徹底 (v0.2 継続)
- 判定: 設計で対処可

#### リスクR2 (新規・最重要): スコア化導入による「重み付け滑り」

- **症状**: スコア式を一度導入すると、「RISING_DECEL+5点」「ADX35超-3点」のような重み付けが追加されていく圧力
- **対策**:
  1. スコア式を**H4×H1 幾何平均のみに固定**(コード上で1行で実装、他の項を追加不可な構造に)
  2. スコア値を**常時表示しない**。縮約モードでのみ表示
  3. スコア値の**色は使わない**(色相は DI バイアスのみ)
  4. **おぱ番人レビュー** をスコア実装段階で必須化
- 判定: 構造的対処で抑制可、ただし**継続監視最重要項目**

#### リスクR3 (新規): D1 背景帯の「環境HOT即エントリー」誤読

- **症状**: D1 BU 帯 + H4 濃青セル が並ぶと「ここは買いだ」と読まれる
- **対策**:
  1. D1 帯は**色を極薄く(不透明度5-15%)** に固定
  2. シグナル行と結果行を**独立に保つ** (v0.2 継続)
  3. 「環境 ≠ シグナル ≠ 結果」の物理的3分離を維持
- 判定: 構造で対処済、リスク低

#### リスクR4 (継続): シグナルマーカー形状の「推奨度」誤読

- 対策: v0.2 継続
- 判定: 凡例設計で対処可

#### リスクR5 (継続): ★評価が個別判断優劣に見える

- 対策: v0.2 継続 (★ = 結果規模、判断良し悪しではない)
- 判定: 凡例で対処可
