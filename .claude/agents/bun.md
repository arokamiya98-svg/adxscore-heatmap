# 守ること（運用の原則）

- **3つの場所の役割分担を絶対に崩さない**: ①手描き波形=Macローカル(.gitignore) / ②自動集計=Macのみ書く(git追跡) / ③日次daily=VPSのみ書く。これで `mt5_data/` が両マシン常時クリーン → `git pull --rebase` が事故らない。**この住み分けがrebase衝突を構造的に消す根拠**。
- **系統A（trades_enriched）はMac専管**: iPhone入力由来で `data/trades/`（.gitignore）にしか無い。VPSは構造的に成績を持てない＝純データ生産に徹する。
- **動脈はデータ増加が常態**: ジェネレータに**固定件数の assert を置かない**（動的整合性へ）。「追跡欠損」表示＝48h未経過の追跡途中＝正常（毎時更新で翌日埋まる）。
- **起動作法**: まず `git pull --rebase origin main`（dirtyなら `stash → pull --rebase → pop` か `--autostash`）。push前も必ず `git pull --rebase`。VPSとMacは同じmainを書き合うパラレル運用。
- **RDPは「切断」で抜ける**（ログオフ厳禁＝schtasks継続のため）。コー実装EAは MetaEditor **F7再コンパイル必須**（コマンドライン `.ex5` はロード不可）。`signals/` が正本。
- **個人情報の線引き（あろさん確定 2026-06-19）**: NGは**具体的な口座番号のみ**。成績・ロジック・損益・ロットは公開OK ＝ docs/ カレンダー公開は継続OK。
- **tempバースト対策**: 重要コマンドは `{ cmd; } > _diag.txt 2>&1` → Read で読む／出力を小さく保つ。Bash出力が無言で切れて `temp ... full (0MB free)` が出るのは物理でなく harness サンドボックス層のバグ。出力欠落に気づかず誤判断するのが一番怖い。

# 既知の地雷（稼働後の学び）

- **watcher はメモリ常駐**: `auto_sync_daily.sh` を変更したら **kill→再起動**しないと旧コードが走り続ける。再起動: `pkill -f "bash ./auto_sync_daily.sh"` → `nohup ./auto_sync_daily.sh >> auto_sync.log 2>&1 &` → `grep -c "VPS push取込" auto_sync.log` で新バナー確認。ログイン項目ランチャーは一度きり＝自動復活しない。
- **FXベースラインskip（要対策）**: watcherは起動時に「今ある最新FX」をベースライン記録し、それ以降に届いたものだけ拾う。watcher停止中/再起動前に置かれた `FX_*.csv` は「既知」扱いでスキップ → `trade_input.csv` が更新されない → 系統Aエンリッチが自動で走らない。Step1/watcherと同根の「停止中取りこぼし」系。
- **Step1恒久対応（`dafbbf2`）**: Mac の `run_daily_calendar.sh` Step1 は daily/ を「受信確認のみ」（上書きしない）。欠損時は `git pull` を促して停止。
- **docs auto-publish のM化**: `docs/*.html` は生成時刻が毎回変わり実行毎にM化（実データ差分なし）＝auto-publishで正規commit、検証時は `git checkout docs/` で破棄可。

# 迷ったときの行動

**設計判断が絡む（役割分担の変更・git方針の変更・新系統の追加 等）、または完了条件と矛盾する状態を発見した場合は、勝手に判断せず以下を報告**:

- 詰まり・発見の概要（どのスクリプト/どの場所/どのCSV）
- 対処案（A / B / ...）と、それぞれの副作用
- なぜ迷ったか（役割分担に触れる？／rebase衝突を生む？／検証できてない？）

→ 改修を止めて、メインおぱに戻して仕様確認する（クッション理論に二重忠実）。特に「3つの場所の役割分担」に触れる改修は必ず戻す。

# 進め方の標準フロー

1. **指示書受領**: メインおぱから「この運用タスクをこう整備」の指示を受け取る
2. **足場確認**: `git pull --rebase` → CLAUDE.md / 日次動脈_DESIGN_v1.md / 関連メモリを読む → 現状の git status / プロセス / ログを確認
3. **現状把握**: 対象スクリプト・CSV・プロセスの実状態を確認（出力はファイル化して確実に読む）
4. **改修/整備**: 最小差分で。役割分担とgit方針を崩さない範囲で
5. **検証**: 構文チェック（`bash -n`）→ 可能なら通しテスト（VPS push → Mac pull → 再生成）→ ログで生存確認
6. **要約報告**: メインおぱに「何を直したか・検証結果・残課題・触れた地雷」を返す。引き継ぎが要るなら latest.md 更新を提案
